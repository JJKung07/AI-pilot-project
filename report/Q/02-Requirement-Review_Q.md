## 1. สรุปผู้บริหาร

รายงานฉบับนี้เป็นการทบทวนความต้องการอย่างเป็นทางการจาก QA ก่อนเริ่มออกแบบ test case โดย RTM จาก Phase 01 ถูก cross-reference กับ Acceptance Criteria และ Business Rules แล้ว ผลสรุปคือ **อนุมัติแบบมีเงื่อนไข (Conditionally Approved)** — มี 2 ช่องว่างสำคัญที่ต้องแก้ไขก่อน sign-off เต็มรูปแบบ

## 3. การทบทวน Acceptance Criteria

| User Story | มี AC? | ผลการตรวจ |
|------------|--------|-----------|
| US-001 | ✅ | เพียงพอ — ครอบคลุม happy path |
| US-002 | ✅ | เพียงพอ — ระบุ error message ของ locked user |
| US-003 | ✅ | มี 2 scenarios (รหัสผ่านผิด + ช่องว่าง) |
| US-004 | ✅ | ตรวจนับ 6 สินค้า |
| US-005–008 | ⚠️ | มีเพียง 2 scenarios สำหรับ 4 stories — ขาด AC สำหรับ Name A-Z และ Price High→Low โดยเฉพาะ |
| US-009 | ✅ | ตรวจ navigation และราคา |
| US-010–012, US-020 | ✅ | พฤติกรรม Add/Remove badge ครอบคลุม |
| US-013–014, US-019 | ✅ | รายการในตะกร้าและตะกร้าว่างครอบคลุม |
| US-015 | ✅ | กรอกข้อมูลถูกต้อง + ขาด postal code |
| US-016 | ✅ | ตัวอย่างคำนวณภาษี ($29.99 → $2.40 → $32.39) |

| US-018 | ✅ | redirect หลัง logout ระบุชัด |

**Acceptance Criteria ที่ขาดหายไป**:
- US-005 (Name A-Z): ไม่มี AC scenario เฉพาะ — ทดสอบทางอ้อมจาก default sort เท่านั้น
- US-008 (Price High→Low): ไม่มี AC scenario ที่ระบุสินค้าแรก/สุดท้ายที่คาดหวัง
- ไม่มี AC สำหรับการลบสินค้าจากหน้ารายละเอียดสินค้า (ช่องว่าง REQ-06)
- ไม่มี AC สำหรับ redirect เมื่อเข้าถึง URL โดยไม่ได้ล็อกอิน (ช่องว่าง REQ-11/SESS-02)

## 5. Edge Cases ที่แนะนำให้เพิ่ม

| พื้นที่ | Edge Case | เหตุผล |
|---------|-----------|--------|
| **CHK-04: การปัดเศษภาษี** | เพิ่มหลายสินค้าที่ 8% ให้ผลเป็นทศนิยมซ้ำ เช่น $15.99 + $15.99 = $31.98 × 0.08 = $2.5584 → ควรปัดเป็น $2.56 | การปัดเศษทางการเงินเป็นจุดบกพร่องที่พบบ่อย ต้องยืนยันว่าปัดขึ้นที่ทศนิยม 2 ตำแหน่ง |
| CHK-04: ภาษีจากสินค้าทั้ง 6 | รวมทุกสินค้า: $129.94 × 0.08 = $10.3952 → $10.40 | ทดสอบการคำนวณที่จำนวนสินค้าสูงสุด |
| AUTH: username ว่างอย่างเดียว | ส่งฟอร์มโดย username ว่างแต่ password ถูกต้อง | ตรวจ error เฉพาะฟิลด์ "Username is required" |
| AUTH: password ว่างอย่างเดียว | ส่งฟอร์มโดย username ถูกต้องแต่ password ว่าง | ตรวจ error เฉพาะฟิลด์ "Password is required" (ยังไม่มีใน AC) |
| CART: เพิ่มทั้ง 6 สินค้าแล้วลบทั้งหมด | Badge ต้องหายไปเมื่อตะกร้าเหลือ 0 | State boundary — ยืนยันว่า badge ซ่อนหรือแสดง "0" |
| PROD-04: ตรวจ default sort | หลังล็อกอินทันที ต้องเป็น Name A-Z โดยไม่ต้องเลือก | Business rule PROD-04 ไม่มี test เฉพาะ |
| CART-03: ตะกร้าคงอยู่ | เพิ่มสินค้า → ไปหน้ารายละเอียด → กลับมาหน้าแคตตาล็อก → badge ยังอยู่ | Business rule CART-03 ต้องตรวจสอบโดยตรง |
| SESS-02: เข้า URL โดยตรง | เข้า `/cart.html`, `/checkout-step-one.html`, `/checkout-step-two.html`, `/checkout-complete.html` โดยไม่มี session | Security boundary — ทุก route ภายในต้อง redirect |
