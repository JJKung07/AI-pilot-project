## สมมติฐาน

- ความยาวสูงสุดของ text fields ทั้งหมด (First Name, Last Name, Postal Code) คือ **255 ตัวอักษร**
- ความยาวต่ำสุดของ required fields คือ **1 ตัวอักษร**
- ค่าที่ต่ำกว่าขั้นต่ำ (min-1) คือ **0 ตัวอักษร (empty string)**

## 3. การประเมินความเสี่ยงของ Boundary Cases

| Boundary Case | ความเสี่ยง | เหตุผล |
|---------------|-----------|--------|
| Empty string (0 ตัวอักษร) สำหรับ required fields | **สูง** | เป็น boundary ที่พบบ่อยที่สุดและมักมี bug — validation ต้องจับได้ |
| max+1 (256 ตัวอักษร) | **ปานกลาง** | อาจทำให้ database overflow หรือ UI แตก หากไม่มี validation ฝั่ง client/server |
| Tax rounding (ทศนิยมซ้ำ) | **สูง** | ข้อผิดพลาดการปัดเศษทางการเงินมีผลกระทบต่อความน่าเชื่อถือและอาจมีปัญหาทางกฎหมาย |
| ตะกร้าว่าง → Checkout | **ปานกลาง** | ระบบควรป้องกันไม่ให้ checkout โดยไม่มีสินค้า หรือแสดงข้อความที่เหมาะสม |
| ตะกร้าเต็ม (6 สินค้า = max) | **ต่ำ** | ระบบจำกัดสินค้าที่ 6 ชิ้นตาม catalog — ไม่สามารถเพิ่มเกินได้ แต่ควรตรวจว่า badge ถูกต้อง |
| สินค้าราคาเท่ากัน (sort stability) | **ปานกลาง** | การเรียงลำดับที่ไม่เสถียรเมื่อราคาเท่ากันอาจทำให้ผู้ใช้สับสน |

---

## 4. สรุป

Boundary cases ที่มีโอกาสเจอ defect สูงสุด:
1. **Empty string (min-1)** สำหรับทุก required field — ต้องมี validation error ที่ชัดเจน
2. **Tax rounding** เมื่อ 8% ให้ผลเป็นทศนิยมมากกว่า 2 ตำแหน่ง — ต้องปัดเศษถูกต้อง
3. **max+1 (256 ตัวอักษร)** — ต้องตรวจว่าระบบจัดการ input ที่เกินขีดจำกัดอย่างเหมาะสม
