## 2. การวิเคราะห์ช่องว่าง

| ช่องว่าง / ความกำกวม | ผลกระทบ | หมายเหตุ |
|----------------------|---------|----------|
| REQ-06: ไม่มี user story สำหรับการลบสินค้าจากหน้ารายละเอียดสินค้า | ปานกลาง | BRD ระบุชัดว่าลบได้จากหน้ารายละเอียด แต่มีเฉพาะ US-012 (catalog) และ US-014 (cart page) |
| REQ-11: ไม่มี user story สำหรับ redirect เมื่อเข้า URL โดยไม่ล็อกอิน | สูง | เป็นข้อกำหนดด้านความปลอดภัย — เข้า /inventory.html หรือ /cart.html โดยไม่มี session ต้อง redirect ไป login |
| REQ-07: BRD ระบุ "จำนวนสินค้าทั้งหมด" vs US-020 ระบุ "unique items added" | ปานกลาง | กำกวมว่า badge นับ product lines หรืออาจนับ quantity ในอนาคต |
| US-013 (ดูตะกร้า) และ US-019 (Continue Shopping) ไม่ได้แมปกับ REQ ใดโดยตรง | ต่ำ | เป็นส่วนหนึ่งของ flow โดยปริยาย แต่ไม่มี REQ อ้างอิงอย่างเป็นทางการ |
| เป้าหมายทางธุรกิจ (ลด cart abandonment, เพิ่ม conversion 15%) ไม่มีเกณฑ์วัดผล | สูง | ไม่สามารถตรวจสอบด้วย functional test ได้หากไม่มี baseline metrics |
| NFR: "Page load < 3 วินาทีบน broadband" — ไม่นิยามความเร็ว broadband หรือเครื่องมือวัด | ปานกลาง | ต้องตกลงเงื่อนไข network และ tooling |
| NFR: WCAG 2.1 AA — ไม่มี browser/device matrix สำหรับ accessibility testing | ปานกลาง | ต้องกำหนดขอบเขตการทดสอบ |

## 3. การจัดประเภทความสามารถในการทดสอบ

### ทดสอบได้

| ข้อกำหนด | เหตุผล |
|----------|--------|
| REQ-01: การยืนยันตัวตน | พฤติกรรมที่สังเกตได้ — ตรวจ login/error messages ผ่าน UI |
| REQ-02: แคตตาล็อกสินค้า | ตรวจได้ — นับจำนวนสินค้า ตรวจฟิลด์ที่แสดง |
| REQ-03: การเรียงลำดับ | ตรวจได้ — เปรียบเทียบลำดับที่แสดงกับที่คาดหวัง |
| REQ-04: หน้ารายละเอียด | ตรวจได้ — navigation และ content assertions |
| REQ-05: ตะกร้า เพิ่ม | ตรวจได้ — สถานะปุ่มเปลี่ยนและ badge เพิ่ม |
| REQ-06: ตะกร้า ลบ | ตรวจได้ — สถานะปุ่มกลับคืนและ badge ลด |
| REQ-07: Badge ตะกร้า | ตรวจได้ — ตัวเลข badge ตรงกับสินค้าในตะกร้า |
| REQ-08: Checkout Step 1 | ตรวจได้ — field validation และ navigation |
| REQ-09: Checkout Step 2 | ตรวจได้ — คำนวณทางคณิตศาสตร์ได้ |
| REQ-10: Checkout Step 3 | ตรวจได้ — ข้อความสำเร็จและตะกร้าว่าง |
| REQ-11: Logout | ตรวจได้ — redirect และ session invalidation |
| NFR: HTTPS traffic | ตรวจได้ — ตรวจ protocol |
| NFR: Page load < 3 วินาที | ตรวจได้ — เมื่อกำหนดสภาพแวดล้อมและเครื่องมือ |
| NFR: Browser compatibility | ตรวจได้ — เมื่อกำหนดเวอร์ชัน browser |

### ทดสอบไม่ได้โดยตรง (ต้องการคำชี้แจง)

| ข้อกำหนด | เหตุผล |
|----------|--------|
| "ลด cart abandonment" | เป็น KPI ทางธุรกิจ ต้องมี analytics baseline ไม่ใช่ pass/fail test |
| "เพิ่ม checkout conversion 15%" | ต้องมีข้อมูลย้อนหลังและช่วงเวลาวัดผล |
| "ทำงานได้แบบ flawless บนทุก browser" | "Flawless" ไม่ใช่เกณฑ์ที่วัดได้ |
| "มอบประสบการณ์ equal or better บน mobile" | เป็นเชิงอัตวิสัย หากไม่มี benchmark ที่กำหนดไว้ |
| "Robust session management and data privacy" | เป็นเป้าหมายที่ดี แต่ต้องแยกเป็น controls ที่ทดสอบได้ |

## 4. การประเมินความเสี่ยง

| พื้นที่เสี่ยง | ความรุนแรง | เหตุผลที่มีความเสี่ยงสูง | จุดเน้นในการทดสอบ |
|--------------|-----------|------------------------|------------------|
| REQ-01: Authentication — **boundary case `locked_out_user`** | Critical | หาก login พัง test อื่นทั้งหมดถูก block; `locked_out_user` เป็น boundary case เฉพาะที่ต้องตรวจอย่างชัดเจน — การอนุญาตให้ locked user เข้าระบบได้เป็นช่องโหว่ด้านความปลอดภัย | ล็อกอินถูกต้อง, ล็อกอินผิด, **locked_out_user error message**, logout, session invalidation |
| REQ-09: Checkout Overview (คำนวณภาษี) | สูง | คำนวณทางการเงิน — ภาษี 8% และการปัดเศษผิดพลาดกระทบความน่าเชื่อถือและอาจมีผลทางกฎหมาย | ยอดรวมย่อยถูกต้อง, ภาษี 8%, พฤติกรรมการปัดเศษ, ยอดรวม = subtotal + tax |
| REQ-11: Session management (เข้าถึงโดยไม่ล็อกอิน) | สูง | ช่องโหว่ความปลอดภัย — หากเข้า internal pages ได้โดยไม่ authenticate ข้อมูลผู้ใช้อาจถูกเปิดเผย | เข้า URL โดยตรง /inventory.html, /cart.html, /checkout-step-one.html โดยไม่ล็อกอิน |
| REQ-06: ลบสินค้าจากทุกจุด | ปานกลาง | ต้อง sync สถานะระหว่างหน้า catalog, detail และ cart — ไม่สอดคล้องทำให้ผู้ใช้สับสน | ลบจากแต่ละหน้า, ตรวจ badge sync, ตรวจสถานะปุ่ม |
| REQ-03: Sorting stability | ปานกลาง | ลำดับผิดอาจซ่อนสินค้าและทำให้การเลือกซื้อสับสน | ทดสอบทั้ง 4 โหมดเรียงลำดับพร้อมตรวจรายการสินค้าทั้งหมด |

## 5. คำถามสำหรับ Business Analyst

1. **REQ-06 (ลบจากหน้ารายละเอียด)**: BRD ระบุว่าลบสินค้าได้จาก "catalog, detail, และ cart pages" แต่ไม่มี user story ครอบคลุมการลบจากหน้ารายละเอียด ควรสร้าง user story ใหม่หรือเจตนาเลื่อนออกไป?

2. **REQ-11 (redirect เมื่อไม่ล็อกอิน)**: หน้าใดบ้างที่นับเป็น "internal pages"? ทุก route ยกเว้นหน้า login ต้อง redirect หรือไม่? หลังล็อกอินแล้วควรกลับไปหน้าที่เคยพยายามเข้าหรือไม่?

3. **REQ-07 (การนับ badge)**: BRD ระบุ "จำนวนสินค้าทั้งหมด" แต่ US-020 ระบุ "unique items added" เนื่องจากระบบปัจจุบันไม่รองรับ quantity > 1 ต่อสินค้า นี่เป็นการออกแบบโดยเจตนาหรือจะเปลี่ยนในอนาคต?

4. **REQ-09 (คำนวณภาษี)**: กฎการปัดเศษสำหรับภาษี 8% คืออะไร? (เช่น ปัดขึ้นที่ทศนิยม 2 ตำแหน่ง?) ภาษีคำนวณต่อรายการหรือจากยอดรวมย่อย?

5. **NFR (Performance/Compatibility)**: มี test matrix ที่ตกลงกันแล้วหรือไม่? (เวอร์ชัน browser เฉพาะ, ประเภทอุปกรณ์, network throttle profiles) และใช้เครื่องมือใดในการวัด performance?

## 6. สรุป

BRD และ User Stories สอดคล้องกันเป็นส่วนใหญ่ — 9 ใน 11 ข้อกำหนดมี user story ครอบคลุมเต็มที่ มี 2 ช่องว่างที่ต้องดำเนินการทันที:

- **REQ-06**: ไม่มี user story สำหรับการลบสินค้าจากหน้ารายละเอียด
- **REQ-11**: ไม่มี user story สำหรับ redirect เมื่อเข้าถึงโดยไม่ล็อกอิน

ความเสี่ยงสูงสุดในการทดสอบคือ **authentication** (โดยเฉพาะ boundary case `locked_out_user` ที่หากจัดการผิดจะ block workflow อื่นทั้งหมด), **การคำนวณภาษี checkout** (ความถูกต้องทางการเงิน) และ **ความปลอดภัยของ session** (การเข้าถึงหน้าโดยไม่ได้รับอนุญาต) เป้าหมายทางธุรกิจและ NFR หลายข้อยังต้องการ baseline และเครื่องมือที่ตกลงกันก่อนจึงจะวัดผลได้
