## 2. การวิเคราะห์ช่องว่าง

| ช่องว่าง / ความกำกวม | ผลกระทบ | หมายเหตุ |
|---|---|---|
| REQ-06 ยังไม่มีการครอบคลุมอย่างชัดเจนสำหรับการลบสินค้าจากหน้ารายละเอียดสินค้า | ปานกลาง | BRD ระบุหน้ารายละเอียดสินค้า แต่ user story ครอบคลุมเฉพาะการลบจากหน้าแคตตาล็อกและหน้าตะกร้า |
| REQ-11 ระบุการจัดการการเข้าถึงหน้า internal page แบบไม่ได้ล็อกอิน แต่ user story ครอบคลุมแค่การ logout | สูง | พฤติกรรมการ redirect เป็นประเด็นด้านความปลอดภัยและควรถูกระบุให้ทดสอบโดยตรง |
| REQ-07 ระบุว่า badge ต้องแสดงจำนวนสินค้าทั้งหมดในตะกร้า แต่ US-020 อธิบายเป็นสินค้าที่เพิ่มแบบ unique items | ปานกลาง | พฤติกรรมปัจจุบันดูเหมือนนับแบบต่อหนึ่งสินค้า แต่กติกาการนับควรยืนยันให้ชัด |
| วัตถุประสงค์ทางธุรกิจ เช่น ลด cart abandonment และเพิ่ม conversion 15% ไม่สามารถวัดได้ด้วย functional test ปกติ | สูง | ต้องมี baseline, ช่วงเวลาวัดผล และเกณฑ์สำเร็จที่ชัดเจนก่อนจะตรวจสอบได้ |
| ข้อความเชิงคุณภาพที่กำกวม เช่น "flawless functionality", "equal or better experience" และ "robust session management and data privacy" ยังไม่สามารถแปลงเป็นเงื่อนไขทดสอบได้ครบ | ปานกลาง | ควรแยกเป็น acceptance criteria ที่เฉพาะเจาะจงและตรวจสอบได้ |
| NFR ยังไม่ระบุวิธี benchmark ที่ชัดเจนสำหรับ performance, mobile experience หรือ accessibility | ปานกลาง | QA ต้องได้ browser/device matrix และวิธีเก็บหลักฐานที่ตกลงร่วมกัน |

## 3. การจัดประเภทความสามารถในการทดสอบ

### ทดสอบได้

| ข้อกำหนด | เหตุผล |
|---|---|
| REQ-01 ถึง REQ-11 | แต่ละข้อเป็นพฤติกรรมที่สังเกตได้จากภายนอก และสามารถตรวจสอบด้วย UI หรือ integration test ได้ |
| Performance: เวลาโหลดหน้าต้องน้อยกว่า 3 วินาทีบน broadband | วัดได้เมื่อกำหนดสภาพแวดล้อมและวิธีจับเวลาอย่างชัดเจน |
| Security: ทราฟฟิกทั้งหมดต้องใช้ HTTPS | ตรวจสอบได้จากระดับ transport |
| Compatibility: Chrome, Firefox, Safari และ Edge | ตรวจสอบได้หากกำหนดเวอร์ชัน browser ให้ชัดตาม latest release |
| Accessibility: WCAG 2.1 AA | ตรวจสอบได้ด้วยเกณฑ์ accessibility และกระบวนการ audit ที่ชัดเจน |
| Sessions ต้องหมดอายุเมื่อ logout | ตรวจสอบได้ด้วยการทดสอบสิทธิ์เข้าถึงหลัง logout |

### ทดสอบไม่ได้โดยตรงหรือยังต้องการคำชี้แจง

| ข้อกำหนด | เหตุผล |
|---|---|
| ลด cart abandonment | เป็น KPI ทางธุรกิจ ไม่ใช่เงื่อนไข functional test โดยตรง |
| เพิ่ม checkout conversion 15% | ต้องใช้ analytics baseline และช่วงเวลาการวัดผล ไม่ใช่การยืนยันแบบ QA ครั้งเดียว |
| ทำงานได้แบบ flawless บนทุก browser | คำว่า "flawless" ยังไม่ใช่เกณฑ์ที่วัดได้ |
| มอบประสบการณ์ equal or better บน mobile web views | คำว่า "equal or better" เป็นเชิงอัตวิสัย หากไม่มี benchmark ที่กำหนดไว้ |
| Secure authentication, robust session management และ data privacy | เป็นเป้าหมายที่ดี แต่ยังต้องแยกเป็น controls ที่ทดสอบได้ชัดเจน |

## 4. การประเมินความเสี่ยง

| พื้นที่เสี่ยง | เหตุผลที่มีความเสี่ยงสูง | จุดเน้นในการทดสอบ |
|---|---|---|
| REQ-01 การยืนยันตัวตน | หากพังจะทำให้การทดสอบส่วนถัดไปทำไม่ได้ และกระทบการควบคุมสิทธิ์เข้าถึงโดยตรง; กรณี locked_out_user เป็น boundary case ที่ต้องตรวจอย่างชัดเจน | ข้อมูลถูกต้อง ข้อมูลผิด ผู้ใช้ที่ถูกล็อก และการ logout/จัดการเซสชัน |
| REQ-09 ยอดรวม checkout | ภาษีและยอดรวมเป็นผลลัพธ์ด้านการเงิน ความผิดพลาดเล็กน้อยส่งผลต่อผู้ใช้และธุรกิจสูง | ยอดรวมย่อย ภาษี 8% การปัดเศษ และยอดรวมสุดท้าย |
| REQ-11 การ logout และการจัดการเซสชัน | ปัญหาด้านความปลอดภัยและการ invalidate เซสชันอาจทำให้เข้าหน้า internal ได้หลัง logout | ขั้นตอน logout, การ redirect การเข้าถึง และการล้างเซสชัน |
| REQ-06 การลบออกจากตะกร้า | สถานะต้องสอดคล้องกันทั้งจากหน้าแคตตาล็อก หน้ารายละเอียด และหน้าตะกร้า | พฤติกรรมการลบจากแต่ละจุดที่รองรับ และการซิงก์ badge |
| REQ-03 การเรียงลำดับ | ความผิดพลาดในการเรียงลำดับอาจซ่อนสินค้า หรือทำให้มุมมองแคตตาล็อกไม่สอดคล้องกัน | โหมดการเรียงลำดับทั้ง 4 แบบ และความเสถียรของลำดับที่แสดง |

## 5. คำถามสำหรับ Business Analyst

1. สำหรับ REQ-06 การลบสินค้าจากหน้ารายละเอียดสินค้าควรถูกยกระดับเป็น requirement ชัดเจนและเพิ่มเป็น user story หรือไม่
2. สำหรับ REQ-11 หน้าใดบ้างที่นับเป็น internal pages สำหรับการตรวจ redirect เมื่อไม่ได้ล็อกอิน และหลัง login แล้วควรกลับไปหน้าที่เคยเปิดอยู่หรือไม่
3. สำหรับ REQ-07 badge ของตะกร้าควรนับเฉพาะสินค้าที่ไม่ซ้ำกัน หรือในอนาคตมีการตีความแบบ quantity-based หรือไม่
4. สำหรับเป้าหมายทางธุรกิจเรื่องลด cart abandonment ด้วยการเพิ่ม checkout conversion QA ควรใช้ baseline metric และช่วงเวลาวัดผลแบบใด
5. สำหรับ performance, compatibility, mobile responsiveness และ accessibility มี test matrix และมาตรฐานหลักฐานสำหรับ sign-off ที่ตกลงกันแล้วหรือไม่

## 6. สรุป

รายการ functional ใน BRD ส่วนใหญ่เชื่อมโยงกับ user story ได้ แต่ยังมีช่องว่างหลัก 2 จุดคือ การลบจากหน้ารายละเอียดสินค้า และการ redirect เมื่อเข้าถึง internal pages แบบไม่ได้ล็อกอิน ขณะเดียวกัน business objectives และ NFR หลายข้อยังต้องการกติกาการวัดผลที่ชัดเจนกว่านี้ก่อนจะใช้เป็นเกณฑ์ผ่าน/ไม่ผ่านได้
